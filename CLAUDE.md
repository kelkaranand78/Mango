## Project Overview

Mango Order Manager ŌĆö a single-page web app for managing mango orders, customers, and delivery lots across seasonal cycles. The entire application lives in one `index.html` file (~3200 lines) with inline CSS + JavaScript. No build tools, no bundler, no package manager.

## Architecture

**Single-file SPA**: All HTML, CSS, and JS are in `index.html`. There is no framework ŌĆö the app uses vanilla JS with string-template rendering.

**State management**: A single global object `S` holds all persistent data (seasons, lots, prices, customers, orders). State is saved to `localStorage` under key `mango_mgr` and optionally synced to Firebase Realtime Database in real-time via `fbRef.on('value', ...)`.

**Rendering**: The `rerender()` function rebuilds the entire `#body` div innerHTML on every state change. It dispatches to tab-specific render functions: `renderDashboard()`, `renderOrders()`, `renderCustomers()`, `renderSetup()`. Focus/cursor position is preserved across re-renders.

**Data model** (all stored in `S`):
- `seasons[]` ŌĆö each has `{id, name}`; one is `activeSeason`
- `lots[seasonId][]` ŌĆö delivery lots with `{id, name, date, color}`
- `lotPrices[lotId]` ŌĆö rate per dozen keyed as `"Type|Size"` (e.g. `"Hapus|Large"`)
- `lotDisabled[lotId]` ŌĆö disabled size combos keyed same way
- `customers[]` ŌĆö `{id, name, custType, flatNo, phone, address}`; types are `Kapil Abhijat` or `Outside KA`
- `orders[seasonId][]` ŌĆö order records with items, totals, payment status
- `prices[seasonId]` ŌĆö legacy/global price data

**Firebase sync**: Config is baked into `BAKED_FB_CONFIG` constant. Firebase arrays are normalized back to JS arrays in `fromFirebase()`. Data versioning uses `_dataVersion` with `migrateData()` for schema evolution.

**Constants**: Mango types (`Hapus`, `Payari`), sizes (`Extra Large` through `Extra Small`), box types (`┬Į Dozen`, `1 Dozen`, `2 Dozen`) are defined at the top of the script block.

## Development

Open `index.html` directly in a browser ŌĆö no server needed. For live reload during development, use any static file server (e.g. `npx serve .` or VS Code Live Server).

External dependencies loaded via CDN:
- Firebase SDK (v10.12.2) ŌĆö realtime database
- xlsx-js-style ŌĆö Excel import/export for customer data
- Google Fonts ŌĆö Playfair Display + DM Sans

## Key Conventions

- All monetary values are in Indian Rupees (Ōé╣), formatted with `toLocaleString('en-IN')`
- Dates use `YYYY-MM-DD` format internally, displayed via `fmtD()` / `fmtS()`
- IDs are 5-char random alphanumeric strings generated by `uid()`
- HTML escaping via `esc()` ŌĆö use it for all user-supplied content in templates
- `saveState()` persists to localStorage AND pushes to Firebase if connected
- The `QR.jpg` file is a payment QR code image used in print labels
- `Firebase.txt` contains the database URL and API key (reference only)
